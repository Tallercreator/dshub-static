# DSHub — документация для разработки

Онлайн-медиа о лучших практиках дизайн-систем в российских цифровых продуктах:
кейсы (интервью и разборы), глоссарий из 121 термина, ресурсы и интерактивные модули.

> **ВАЖНО для Code-фазы.** Это **новый проект на новом стеке** — отдельная папка/репозиторий.
> Старая реализация на Ruby on Rails не переносится и не используется. Начинаем с нуля,
> опираясь на эти доки и на дизайн в Figma.

## Источник правды

## Рекомендуемый стек (подтвердить)
Главный канал привлечения у DSHub — **поисковый трафик на страницы терминов и кейсов** (см. `06-research.md`),
поэтому SEO/серверный рендер критичны. SPA на чистом React для этого слабоват.

- **Astro + React-острова** — SSG/SSR из коробки (отличный SEO), контент как коллекции (JSON/MD),
  интерактивные модули (Песочница токенов, Анатомия) подключаются React-островами. Деплой: Vercel / Netlify.
- Альтернатива — **Next.js** (App Router, SSG/SSR) или Vite + React с пререндером/SSG ради SEO.

**Стилизация — без Tailwind.** Мы строим настоящую DS по UDF, где компонент владеет своими стилями и
ссылается на семантические токены. Utility-first под это не подходит: он размазывает стиль по разметке и
дублирует уже существующую токен-систему. Поэтому:
- токены → `src/styles/tokens.css` (CSS-переменные, = Figma Variables один-в-один);
- стили — **на уровне компонента** (CSS Modules: `A_Button.module.css`), потребляют переменные напрямую;
- `hover`/`pressed` — через CSS (`:hover`, `:active`), а не пропсы; пропсы только смысловые (`value`, `variant`, `disabled`);
- контент → JSON; рендер статей/терминов — `ContentRenderer` по типу блока.

### Эталон стиля компонента
```css
/* tokens.css — = Figma Variables */
:root{
  --surface-accent:#FF4D63; --surface-accent-hover:#E63A50;
  --text-on-accent:#000; --radius-md:12px; --space-lg:12px; --space-xl:16px;
}
```
```css
/* A_Button.module.css */
.button{
  background:var(--surface-accent); color:var(--text-on-accent);
  border-radius:var(--radius-md); padding:var(--space-lg) var(--space-xl);
}
.button:hover{ background:var(--surface-accent-hover); }
```
Стиль лежит вместе с компонентом, назван по UDF, привязан к семантике — сквозной нейминг Figma → код сохраняется,
без промежуточного конфига.

## Архитектура
- **UDF-компоненты** (`02-udf-system.md`): слои `Q_ → A_ → M_ → O_ → W_ → S_ → T_`, строго снизу вверх.
- **Контент-driven** (`04-content-model.md`): кейсы и термины — это JSON-массивы типизированных блоков,
  которые рендерит `ContentRenderer` (switch по `type` → компонент). Глоссарий — по полю `visualType`.
- **Токены** (`01-design-tokens.md`): core (примитивы) + semantic (роли). В коде = CSS-переменные.

## Предлагаемая структура папок
```
dshub/
├─ docs/                      # эти md — источник правды
├─ src/
│  ├─ styles/tokens.css       # из 01-design-tokens.md
│  ├─ components/
│  │  ├─ quarks/ atoms/ molecules/ organisms/ wrappers/ superorganisms/   # каждый компонент + свой Component.module.css
│  ├─ content/                # JSON: cases/ terms/ resources/
│  ├─ lib/ContentRenderer.*    # рендер блоков по type
│  └─ pages|routes/           # home, cases, case, glossary, term, resources
└─ public/
```

## Порядок работы
1. `tokens.css` из `01-design-tokens.md` → подключить глобально (CSS-переменные).
2. Компоненты по `03-build-checklist.md` — строго снизу вверх (квАрки → атомы → … → шаблоны). Стиль каждого — в `Component.module.css` рядом с ним.
3. `ContentRenderer` + схемы из `04-content-model.md`; наполнить парой кейсов/терминов как seed.
4. Собрать страницы из готовых компонентов.

## Объём v1 (MVP) и что отложено
- **v1:** токены, ядро компонентов, главная + страница кейса + страница термина + словарь (десктоп).
- **Вторая волна:** адаптивы (mobile/tablet), интерактивные модули (Песочница, Анатомия), демо-виджеты глоссария, страницы Ресурсы/О проекте.
